# 机票列表项目 - 开发规范文档

## 一、文档说明

本文档旨在规范项目开发流程、代码质量和协作方式，确保项目的可维护性和稳定性。所有开发人员（包括AI助手）必须严格遵守本规范。

---

## 二、Git 管理规范

### 2.1 分支结构

| 分支类型 | 命名规范 | 用途说明 |
|----------|----------|----------|
| `main` | - | 主分支，存放稳定发布版本，禁止直接提交 |
| `develop` | - | 开发分支，日常开发的基础分支 |
| `feature/*` | `feature/功能名称` | 功能开发分支，基于 develop 创建 |
| `bugfix/*` | `bugfix/问题描述` | Bug修复分支，基于 develop 创建 |
| `hotfix/*` | `hotfix/紧急修复` | 紧急修复分支，基于 main 创建 |

### 2.2 分支操作流程

#### 2.2.1 开发新功能

```bash
# 1. 切换到 develop 分支并拉取最新代码
git checkout develop
git pull origin develop

# 2. 创建功能分支（功能名称使用小写，多个单词用连字符）
git checkout -b feature/implement-backend-api

# 3. 开发过程中定期提交
git add .
git commit -m "feat: 描述功能点"

# 4. 推送到远程功能分支
git push -u origin feature/implement-backend-api

# 5. 在 GitHub 上创建 Pull Request，目标分支为 develop
# 6. 代码审查通过后合并到 develop

# 7. 本地更新并删除功能分支
git checkout develop
git pull origin develop
git branch -d feature/implement-backend-api
```

#### 2.2.2 修复 Bug

```bash
git checkout develop
git pull origin develop
git checkout -b bugfix/fix-date-scroll-center
# ... 修复代码 ...
git add .
git commit -m "fix: 修复日期滚动居中问题"
git push -u origin bugfix/fix-date-scroll-center
# 创建 PR 到 develop
```

#### 2.2.3 紧急修复（Hotfix）

```bash
git checkout main
git pull origin main
git checkout -b hotfix/fix-critical-error
# ... 修复代码 ...
git add .
git commit -m "hotfix: 修复紧急问题"
git push -u origin hotfix/fix-critical-error
# 创建 PR 到 main
# 同时需要合并到 develop
git checkout develop
git merge hotfix/fix-critical-error
git push origin develop
```

### 2.3 提交规范

**提交信息格式**：`类型: 描述`

| 类型 | 说明 | 示例 |
|------|------|------|
| `feat` | 新增功能 | `feat: 添加日期选择组件` |
| `fix` | 修复 Bug | `fix: 修复骨架屏加载闪烁` |
| `docs` | 文档更新 | `docs: 更新项目计划文档` |
| `style` | 样式调整 | `style: 优化航班卡片间距` |
| `refactor` | 代码重构 | `refactor: 重构状态管理逻辑` |
| `test` | 测试相关 | `test: 添加单元测试` |
| `chore` | 构建/工具 | `chore: 更新依赖版本` |

**示例**：
```
feat: 实现后端Express服务
- 创建mock数据模块
- 定义航班查询API接口
- 配置CORS跨域
```

### 2.4 代码审查要求

1. 所有功能分支必须创建 Pull Request 到 develop
2. PR 标题清晰描述改动内容
3. PR 描述包含：改动目的、关键实现、测试方法
4. 至少经过一次代码审查才能合并
5. 合并时使用 Squash Merge 保持提交历史整洁

## 三、开发规范

### 3.1 任务执行流程

#### 3.1.1 接收任务

1. 仔细阅读用户需求，明确任务范围
2. 分析需求是否清晰，若有疑问及时询问用户
3. 确认技术方案和实现方式

#### 3.1.2 制定计划

1. 使用 `TodoWrite` 工具创建任务清单
2. 任务清单需包含：具体步骤、优先级、预计耗时
3. 若计划有变动，需更新 `项目计划.txt` 文档

#### 3.1.3 实施开发

1. 严格按照计划执行，每完成一个任务更新状态
2. 代码实现遵循技术栈规范（React + TypeScript + SCSS）
3. 组件拆分合理，职责单一
4. 代码注释清晰，关键逻辑必须有注释
5. 及时更新相关文档

#### 3.1.4 测试验证

1. 功能开发完成后进行构建测试
2. 启动开发服务器验证页面效果
3. 检查控制台是否有错误或警告
4. 确保所有功能正常运行

#### 3.1.5 提交代码

1. 编写规范的提交信息
2. 确保 `.gitignore` 已正确配置
3. 推送代码到远程仓库

### 3.2 文档管理规范

#### 3.2.1 项目计划文档

- 文件路径：`项目计划.txt`
- 更新时机：
  - 项目初始化时创建
  - 每次开发任务开始前更新任务状态
  - 计划变更时及时修改
- 内容要求：
  - 明确的阶段划分
  - 每个阶段的任务清单
  - 任务完成状态标记（✅ 已完成、⏳ 进行中）
  - 项目当前状态说明

#### 3.2.2 接口文档

- 文件路径：`server/api-docs.md`（待创建）
- 更新时机：新增或修改 API 接口时
- 内容要求：
  - 接口路径和 HTTP 方法
  - 入参说明（参数名、类型、必填、描述）
  - 出参说明（字段名、类型、描述）
  - 错误码说明

#### 3.2.3 技术文档

- 记录技术选型依据
- 记录遇到的困难和解决方案
- 记录优化思路和改进方案

### 3.3 代码规范

#### 3.3.1 TypeScript 规范

1. **类型定义**：所有变量、函数、组件必须有明确的类型定义
2. **接口定义**：复杂数据结构使用 interface 定义
3. **泛型使用**：需要复用的组件/函数使用泛型
4. **严格模式**：启用 strict 模式，禁止 any 类型

#### 3.3.2 React 规范

1. **组件命名**：PascalCase，文件名与组件名一致
2. **函数组件**：优先使用函数组件 + Hooks
3. **状态管理**：根据场景选择 MobX 或 Redux
4. **Props 传递**：使用 interface 定义 Props 类型
5. **条件渲染**：使用 && 或三元表达式
6. **列表渲染**：必须添加 key 属性

#### 3.3.3 SCSS 规范

1. **命名规范**：BEM 命名法（block__element--modifier）
2. **文件结构**：每个组件独立 SCSS 文件
3. **变量使用**：颜色、字体大小等使用变量定义
4. **嵌套层级**：不超过 3 层
5. **避免 !important**：除非必要，禁止使用

#### 3.3.4 目录结构规范

```
client/src/
├── components/           # 组件目录
│   ├── ComponentName/   # 组件文件夹（大驼峰）
│   │   ├── index.tsx    # 组件主文件
│   │   └── index.scss   # 组件样式
├── store/               # 状态管理
│   ├── mobx/            # MobX 实现
│   └── redux/           # Redux 实现
├── services/            # API 服务
├── types/               # TypeScript 类型定义
├── utils/               # 工具函数
├── App.tsx              # 主应用组件
├── index.tsx            # 入口文件
└── index.scss           # 全局样式
```

## 四、质量保证规范

### 4.1 构建检查

1. 每次提交前必须执行构建命令
2. 构建成功才能推送代码
3. 禁止忽略构建错误

### 4.2 错误处理

1. 前端：使用 try-catch 捕获异常
2. 后端：统一错误处理中间件
3. API 请求：处理请求失败和超时情况

### 4.3 性能优化

1. 组件懒加载（按需加载）
2. 列表虚拟化（长列表优化）
3. 图片优化（懒加载、压缩）
4. 缓存策略（localStorage、请求缓存）

### 4.4 代码质量检查

1. 无未使用的变量和导入
2. 无控制台警告
3. 代码逻辑清晰，无冗余代码
4. 组件拆分合理，无巨型组件

## 五、协作规范

### 5.1 沟通规范

1. 每日汇报任务进度
2. 遇到阻塞问题及时反馈
3. 需求变更时确认影响范围
4. 保持沟通渠道畅通

### 5.2 版本控制

1. 定期推送代码到远程仓库
2. 避免长时间不提交
3. 提交信息清晰描述改动
4. 合并前解决冲突

### 5.3 安全规范

1. 禁止提交敏感信息（密码、密钥等）
2. API 接口添加必要的参数校验
3. 注意 XSS 和 CSRF 防护
4. 使用 HTTPS 协议
